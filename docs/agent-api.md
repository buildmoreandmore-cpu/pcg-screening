# PCG Agent API

Single-tenant admin API for the PCG Screening AI agent (Gwen).

**Base URL (prod):** `https://pcg-screening.vercel.app/api/agent`
**Base URL (local):** `http://localhost:3000/api/agent`

The agent is the only consumer. There is no per-user scoping — calls with a valid key see everything across every employer client.

---

## Authentication

Every request must include:

```
Authorization: Bearer <PCG_AGENT_API_KEY>
```

- `PCG_AGENT_API_KEY` is a 32-byte hex secret generated by `node scripts/generate-agent-keys.mjs`.
- Missing or invalid key → `401 Unauthorized`.
- Missing server-side env var → `500` (tells the agent PCG is misconfigured, not that auth failed).

## Endpoints

All responses are JSON. Timestamps are ISO 8601 UTC. Shapes are consistent across list/detail endpoints.

### `GET /candidates`

List candidates across all clients, newest first.

**Query params**

| Param  | Type   | Notes |
|--------|--------|-------|
| status | string | Filter by candidate status enum (see below) |
| since  | ISO 8601 | Only candidates created at or after this timestamp |
| limit  | number | Default 100, max 500 |

**Example**

curl -H "Authorization: Bearer $KEY" \
  "$BASE/candidates?status=in_progress&limit=50"

## Status Enums

### Candidate `status`
- `submitted` — invite created, candidate hasn't completed intake
- `in_progress` — payment received, screening running
- `completed` — screening finished, report available
- `cancelled` — manually cancelled by admin

### Candidate `payment_status`
- `pending` · `paid` · `refunded` · `expired`

### Candidate `consent_status`
- `pending` · `signed` · `declined`

### Screening `status`
Same as candidate status; screenings only surface rows in `in_progress`, `completed`, `cancelled`.

### Package types
Packages are per-client and stored as a `jsonb` array on `clients.packages`. Default catalog:

- `Basic Background Check` — SSN trace + national criminal
- `Standard Background Check` — Basic + county criminal + sex offender registry
- `Premium Background Check` — Standard + employment + education verification
- `Drug Test Add-On` — 10-panel drug screening
- `Custom Screening` — ad hoc, price set per candidate

Treat `package_name` as a free-form string — don't hardcode the list.

### Billing types
`immediate` · `net_30` · `net_60` · `net_90`

## Webhook Events

PCG POSTs to `PCG_AGENT_WEBHOOK_URL` on key events. Dispatch is **fire-and-forget** — webhook failures do not block the user-facing action.

### Headers

```
Content-Type: application/json
X-PCG-Signature: <hmac-sha256 hex of the raw body, using PCG_AGENT_WEBHOOK_SECRET>
X-PCG-Event-Type: <event_type>
```

### Payload envelope

```json
{
  "event_type": "candidate.submitted",
  "summary": "Jane Doe invited for Standard Background Check (Acme Corp)",
  "detail": { /* event-specific fields */ },
  "timestamp": "2026-04-09T12:00:00Z"
}
```

### Event types

| `event_type` | When it fires | `detail` fields |
|---|---|---|
| `candidate.submitted` | Employer invites a candidate (portal or manual) | `tracking_code`, `client_id`, `client_name`, `candidate_email`, `package_name`, `source` |
| `candidate.status_changed` | Admin changes a candidate status OR Stripe payment moves `submitted → in_progress` | `candidate_id`, `candidate_name`, `client_id`, `client_name`, `previous_status`, `new_status`, `notes`, `changed_by` \| `trigger` |
| `screening.completed` | Status transitions to `completed` | `candidate_id`, `candidate_name`, `client_id`, `client_name`, `package_name`, `tracking_code` |
| `screening.flagged` | Candidate with `sla_flagged=true` gets a status update | `candidate_id`, `candidate_name`, `client_id`, `client_name`, `reason` |
| `client.created` | New employer client onboarded via admin | `client_id`, `client_name`, `slug`, `contact_name`, `contact_email`, `billing_type` |
| `payment.received` | Stripe `checkout.session.completed` | `candidate_id`, `candidate_name`, `tracking_code`, `amount`, `currency`, `stripe_session_id`, `package_name`, `client_slug` |
| `payment.failed` | Stripe `checkout.session.expired` | `candidate_id`, `stripe_session_id`, `reason` |

When a status changes to `completed`, PCG fires **two** events: `candidate.status_changed` and `screening.completed`. Dedupe by `candidate_id + timestamp` if you need a single notification.

When a Stripe payment completes, PCG fires **two** events: `payment.received` and `candidate.status_changed`.

## HMAC signature verification (Node)

```js
import { createHmac, timingSafeEqual } from 'node:crypto'
import express from 'express'

const app = express()

// Capture raw body for HMAC verification
app.use(express.json({
  verify: (req, _res, buf) => { req.rawBody = buf }
}))

app.post('/webhooks/pcg', (req, res) => {
  const secret = process.env.PCG_AGENT_WEBHOOK_SECRET
  const signature = req.header('X-PCG-Signature') || ''
  const expected = createHmac('sha256', secret).update(req.rawBody).digest('hex')

  const sigBuf = Buffer.from(signature, 'hex')
  const expBuf = Buffer.from(expected, 'hex')

  if (sigBuf.length !== expBuf.length || !timingSafeEqual(sigBuf, expBuf)) {
    return res.status(401).send('invalid signature')
  }

  const { event_type, summary, detail, timestamp } = req.body
  console.log('[pcg]', event_type, summary)

  // Handle event...
  res.status(200).send('ok')
})

app.listen(3001)
```

The signature is computed over the **raw request body** — not over a re-serialized JSON object. Frameworks that auto-parse JSON must expose the raw bytes (like the `verify` hook above), or you'll get signature mismatches from whitespace differences.

## Environment variables

| Var | Required | Notes |
|---|---|---|
| `PCG_AGENT_API_KEY` | yes | 32-byte hex, set via `scripts/generate-agent-keys.mjs` |
| `PCG_AGENT_WEBHOOK_URL` | no (but webhook is disabled without it) | Full URL of the agent's webhook endpoint |
| `PCG_AGENT_WEBHOOK_SECRET` | no (but webhook is disabled without it) | 32-byte hex used for HMAC signing |

When `PCG_AGENT_WEBHOOK_URL` or `PCG_AGENT_WEBHOOK_SECRET` is unset, dispatches silently no-op.

## Generating credentials

```bash
node scripts/generate-agent-keys.mjs
```

Prints fresh `PCG_AGENT_API_KEY` and `PCG_AGENT_WEBHOOK_SECRET` values once. Copy them into `.env.local` and the Vercel project env — they are not stored anywhere by the script.
